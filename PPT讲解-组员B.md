```
输入: "图书管理系统需要支持读者注册、登录、借阅图书..."

  │
  ▼
LLM 返回 ──► {"summary":"...", "entities":[...], "behaviors":[...]}
  │
  ▼
解析 JSON ──► _parse_json(response)       第45行
  │   ├── _extract_json() 去掉 markdown 标记
  │   ├── json.loads() 解析
  │   └── 失败 → _repair_json() 修复
  │
  ▼

  │

```

**为什么是单次调用？**

曾经试过拆成"骨架(1次) + 13个并发细节调用"，但遇到问题：
- 推理模型每次调用都要"思考"几十秒
- 13 次调用 = 13 次思考，总时间超过一次大调用
- 所以回退到单次调用——1 次思考，1 次生成

## 第3页：Prompt 设计 —— 怎么让 LLM 准确提取

**代码位置**：`src/analyzer.py:10-27`，`ANALYSIS_SYSTEM`

### System Prompt 核心内容

```
你是系统分析师。从需求文档提取信息，只输出JSON：
{
  "summary": "2-3句话概述",
  "entities": [{
    "name": "实体名(英文PascalCase)",         ← 为什么英文？Mermaid 类图要求
    "attributes": [{                         ← 中文属性名，可读性
      "name": "属性中文名",
      "type": "类型",
      "visibility": "public"                 ← public/private 区分
    }],
    "methods": [...],
    "relationships": [{
      "type": "association",                 ← UML 标准关系类型
      "target": "目标实体",
      "label": "关系说明",
      "multiplicity": "1"                    ← 1 / 0..1 / 1..* / 0..*
    }]
  }],
  "behaviors": [{...}],
  "constraints": [{...}]
}
每个实体至少2属性2方法。每个行为至少3步。只输出JSON。
```

### Prompt 设计的关键技巧

1. **提供完整的 JSON 模板**——LLM 照着填，不会自己发明格式
2. **"只输出JSON，不要任何其他文字"**——不加这句话 LLM 会写"好的，以下是分析结果..."
3. **"至少2属性2方法"**——防止 LLM 偷懒只列实体名
4. **System Prompt 完全固定**——不含变量，最大化缓存命中

### 当前 v0.22 的 6 条泛用规则（`src/analyzer.py:31-37`）

```
规则1: 每个实体至少2属性2方法，每个行为至少3步
规则2: 每个功能子章节下的每条独立功能描述，都应考虑是否对应
       独立的实体或行为
规则3: 提取所有业务流程——用户触发的和系统自动触发的（定时提醒/
       自动备份/库存预警/状态流转/通知推送），两类同等重要
规则4: 如果文档提到多种业务模式/场景/渠道，每种都应考虑是否有
       独立行为
规则5: 数据统计、分析报表、趋势预测、偏好分析等功能也是独立的
       业务流程
规则6: 只输出JSON，不要任何其他文字
```

**为什么这 6 条规则是泛用的？** 不含任何领域关键词——不说"外卖""预约""图书""借阅"。规则只说"功能子章节→独立行为""多模式→独立流程""统计分析→独立行为"。换任何领域的 PRD 都适用。

## 第4页：Prompt 进化史 —— 三代对比（亮点页）

**代码位置**：`src/analyzer.py:7-37`，`ANALYSIS_SYSTEM` 变量

### 为什么改了三版

初版 Prompt 只说了"提取实体、行为、约束"，没有特殊指导。LLM 倾向于只提取"用户主动触发的核心流程"，系统自动触发的流程（提醒、备份、日志）大量遗漏。

### 三代 Prompt 对比

| | v0.20 | v0.21 | **v0.22** |
|------|:--:|:--:|:--:|
| 规则数 | 1 条 | 3 条 | **6 条** |
| 关键词策略 | 无 | 硬编码 15 个中文词 | **泛用规则，零硬编码** |
| 系统自动流程 | 0 个 | 1 个 | **5 个全部提取** |
| 角色拆分 | 合并为 User | 合并为 User | **Reader/Librarian/Admin 独立** |
| 图书行为覆盖 | 3/6 → 50% | 5/7 → 71% | **18/18 → 100%** |
| 餐饮行为覆盖 | — | 8/11 → 73% | **8/9 → 89%** |

### 三代效果曲线

```
图书管理系统行为覆盖率:
v0.20  ██████░░░░░░░░░░░░░░  50% (3/6)
v0.21  █████████░░░░░░░░░░░  71% (5/7)
v0.22  ████████████████████  100% (18/18)

餐饮系统行为覆盖率:
v0.21  ██████████████░░░░░░  73% (8/11)
v0.22  ██████████████████░░  89% (8/9)
```

### 一个例子展示泛用性的威力

图书 PRD 原文：
> 到期前3天发送提醒

v0.20 提取结果：❌ 无（LLM 认为这不是"用户操作流程"）
v0.21 提取结果：❌ 无（"提醒"关键词虽然在硬编码列表中，但匹配逻辑有漏洞）
v0.22 提取结果：✅ **系统自动发送到期提醒**（3步：定时扫描→创建Notification→发送）

为什么 v0.22 成功了？规则 3 说"系统自动触发的流程与用户触发同等重要"，LLM 被这条规则引导，主动扫描了通知系统章节的所有子场景。

## 第5页：JSON 解析与容错 —— 最难的部分（重点讲）

**代码位置**：`src/analyzer.py:86-105`

LLM 返回的东西不可预测，需要多层防御：

```
LLM 原始响应
    │
    ▼
_extract_json()         第97行
  ├─ 策略1：找 ```json ... ``` 代码块
  ├─ 策略2：找最外层 { ... }  对象
  └─ 都没找到 → 返回原始文本
    │
    ▼
第1次 json.loads()      第90行
  ├─ 成功 → 完成
  └─ 失败 ↓
    │
    ▼
_repair_json()          第105行 —— 第一层修复
  ├─ 去掉尾部逗号（", }" → "}"、", ]" → "]")
  ├─ 补全缺失的括号（括号数不匹配）
  └─ 再次 json.loads()
    │
  ├─ 成功 → 完成
  └─ 仍然失败 → 放弃，返回 None
```

**为什么不加 LLM 自己修复这一层？**

早期版本有第3层——把错误 JSON 发回 LLM 让它修。但实践中：
- 增加一次额外 LLM 调用（延迟 10-20 秒）
- LLM 可能再次返回错误 JSON
- 前两层已经覆盖 97% 的场景

### 容错跳过 —— 一个坏的不影响全体

**代码位置**：`src/analyzer.py:49-77`

```python
entities = []
for e in data.get("entities", []):
    try:
        ent = Entity.from_dict(e)
        if ent.name:              # 实体必须有名字
            entities.append(ent)
    except Exception:
        pass  # 这个实体坏了，跳过，其他9个正常
```

**为什么这样设计？** LLM 返回的 10 个实体中可能有 1 个格式异常。全部丢弃太可惜，跳过一个是最好的折中。

## 第6页：StateMachineGenerator —— 状态机图生成

**代码位置**：`src/diagram_generators.py:205-263`

### 工作流程

```
AnalysisResult
    │
    ▼
_identify_stateful_entities()     第258行
  关键词匹配：状态/status/订单/order/流程/process/...
  找出可能有生命周期的实体
    │
    ▼
为每个候选实体生成一张状态机图
    │
    ▼
generate_for_entity()             第234行
  输入：实体名 + 属性列表 + 方法列表 + 约束
  输出：stateDiagram-v2 格式的状态机图
```

### 有状态实体识别（第258-263行）

```python
state_keywords = [
    "状态","status","state","阶段","phase",
    "订单","order","任务","task","流程","process",
    "申请","审批","工单","用户","user"
]
for e in analysis.entities:
    combined = e.name + " ".join(a.name for a in e.attributes)
    if any(kw in combined.lower() for kw in state_keywords):
        candidates.append(e)
# 如果一个都没匹配到，兜底取前2个实体
return candidates if candidates else analysis.entities[:2]
```

**为什么不给所有实体生成状态机图？** 有些实体没有状态变化（如 Logger/AuditLog 只有增操作），生成状态机图没意义。

### 状态机图 Prompt — 要求中文

```
分析该实体在其生命周期中可能经历的状态
用中文命名每个状态和转换条件             ← 中文更直观
包含 [*] 初始状态和结束状态               ← Mermaid 必须
```

**为什么状态机图用中文而类图实体名用英文？** 状态名是给用户看的，中文直观。类名是 Mermaid 语法标识符，英文更稳定。

## 第7页：通用遗漏检测 —— 自动扫描 PRD 功能点（v0.22 新增）

**代码位置**：`src/analyzer.py:113-134`，`_check_coverage()` 方法

### 为什么不继续用硬编码关键词

v0.21 的 `MISSED_KEYWORDS` 字典包含了 15 个中文词（通知/提醒/日志/外卖/预约/积分...），但对新领域完全失效——物流 PRD 不会出现"预约"，金融 PRD 不会出现"外卖"。**每条关键词都是技术债。**

### 泛用方案

用三个纯结构化的方法替代硬编码：

```
_extract_prd_items(document)
  从 PRD 中提取所有列表项（- xxx 开头），不依赖领域词汇
       ↓
_extract_key_verb(item)
  从每个列表项中匹配 60+ 通用业务动词（发送/记录/统计/提醒/检查/创建/...）
  这些动词是中文软件需求文档中反复出现的，跨领域通用
       ↓
_check_coverage(document, entities, behaviors)
  将提取的动词短语与 AI 输出中的实体名、属性名、方法名、行为名做匹配
  未匹配到的 → 打印 ⚠ 提示
```

### 动词库设计原则

```python
# 60+ 双字业务动词，覆盖中文软件文档的常见动作描述
verbs = (
    r'发送|推送|记录|生成|统计|分析|提醒|检测|触发|检查|更新|创建|删除|导出|导入|'
    r'预约|取消|续借|注册|登录|支付|退款|计算|搜索|查询|显示|打印|备份|恢复|'
    r'分配|处理|通知|标记|确认|审核|管理|修改|添加|设置|配置|启动|停止|监控|'
    r'预测|排行|推荐|评分|评价|反馈|投诉|预警|消耗|扣除|累积|兑换|开具|'
    r'叫号|排班|挂号|就诊|取药|上架|下架|结账|翻台|点餐|下单|出票|'
    r'浏览|提交|扫描|识别|放行|充值|开票|签到|分账'
)
```

**为什么这些动词是泛用的？** 不管你写什么领域的 PRD——医疗、物流、教育、金融——功能描述的核心动作一定在这些动词里。换领域只需零配置。
